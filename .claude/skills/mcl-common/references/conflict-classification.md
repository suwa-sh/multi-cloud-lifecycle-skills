# 競合分類

## 目的
canonical model を複数のクラウドベンダーにマッピングする際に生じる競合を分類・管理する。

## 競合タイプ

### 1. capability 競合
ベンダー間で同じ capability に対するアプローチが根本的に異なる。
- **例**: AWS SCP のデニーリスト方式 vs Azure Policy のデニー/監査モード
- **解決**: 全ベンダーに対応できる canonical model を選択する。最小公倍数とベンダー固有拡張を記録する

### 2. granularity 競合
同じ関心事に対しベンダーがサポートする粒度が異なる。
- **例**: AWS のアカウントレベル分離 vs Azure のサブスクリプション + リソースグループ
- **解決**: 利用可能な最も細かい粒度でモデル化する。粗いベンダーには明示的な注記でマッピングする

### 3. feature gap
一部のベンダーには他が提供する capability がネイティブに存在しない。
- **例**: 2 つのクラウドにはある特定のコンプライアンス機能が 3 つ目にはない
- **解決**: ギャップとして記録する。ワークアラウンドまたは deviation record を必須とする

### 4. naming 競合
同じ概念に対しベンダー間で異なる用語を使用する。
- **例**: AWS "Security Group" vs Azure "NSG" vs GCP "Firewall Rule"
- **解決**: canonical 名を使用する。用語マッピングテーブルを維持する

### 5. behavioral 競合
同じ機能名だがデフォルト動作が異なる。
- **例**: クラウド間でデフォルトの暗号化設定が異なる
- **解決**: 動作の違いを記録する。implementation spec で明示的に設定を指定する

## Decision Record の要件
すべての競合解決は以下を含む `decision_record` を生成すること:
- 競合タイプと説明
- 検討した選択肢
- 決定内容と根拠
- リスク評価
- 影響を受ける canonical 要素
- レビュー日（決定を見直すべき時期）
